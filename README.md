# P25 — Two-Stage Crime Risk Intensity & Emergency Activation Pipeline

> **ML Systems Engineering Capstone Project**
> A two-dataset, two-stage, 16-model cascade system for municipal emergency dispatch optimisation.

---

## Project Summary

This project builds a two-stage machine learning pipeline that:

1. **Stage 1** — Predicts the *Crime Risk Intensity Score* of a US community using demographic data (4 regression models competing)
2. **Stage 2** — Predicts the *probability of Level-1 Emergency Activation* using Stage 1's output + state-level crime context (4 classifiers competing)

The final output is `P(EmergencyActivation)` — a continuous probability score that feeds a tiered dispatch decision system.

## Pipeline Architecture

```
data/raw/communities.data  (D1 — 1994 communities, 100 features)
         │
         ▼
  STAGE 1 MODEL ZOO
  LinearRegression  →  LinReg_Y1hat
  Ridge             →  Ridge_Y1hat
  RandomForest      →  RF_Y1hat
  GradientBoosting  →  GBM_Y1hat
         │
  STACKING LAYER (Option B)
  Append all 4 Y1_hat columns to D2 features
         │
data/raw/state_crime.csv   (D2 — 52 states, 7 crime rate features)
         │
         ▼
  STAGE 2 MODEL ZOO  (trains on D2 + 4×Y1_hat = 11 features)
  LogisticRegression
  DecisionTree
  RandomForest
  GradientBoosting
         │
         ▼ Best of 16 combinations selected by AUC
         │
  P(EmergencyActivation) ∈ [0, 1]
         │
  Business threshold layer:
  P >= 0.80  →  Level-1: Full deployment
  P >= 0.50  →  Level-2: Standby + monitor
  P <  0.50  →  Standard patrol
```

## Why Not Neural Networks?

- Dataset size: 1,994 rows. Neural networks need >> 10,000+ rows for 100 features
- Feature/row ratio: 0.05 (5× above safe NN threshold of 0.01)
- Empirical: LR R²=0.649 > RF R²=0.637 — data is predominantly linear
- Interpretability: emergency dispatch decisions must be explainable
- Tabular data rule: GBM consistently outperforms NN on tabular datasets < 50K rows

## Key Design Decisions

| Decision | Reasoning |
|----------|-----------|
| Binary Y2, not 3-class | Y2 derived from one threshold on Y1. Three dispatch tiers are a post-prediction business layer, not model classes. |
| Stage 2 exists despite Stage 1 | D2 state features add context Stage 1 cannot see: same Y1_hat in DC vs North Dakota produces different correct dispatch decisions. |
| Option B stacking | All 4 Y1_hat columns appended: disagreement between models is informative signal for Stage 2. |
| log1p on Y1 | Y1 skew=1.52. Transform applied before LR/Ridge only. RF/GBM train on raw Y1. |
| D2 filtered to 1995 | Matches FBI UCR year of D1. Temporal alignment ensures state context and community data describe the same period. |
